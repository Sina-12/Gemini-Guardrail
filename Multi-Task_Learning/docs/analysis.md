# Analysis

This file will be used for the detailed analysis part of Sprint 3.

The main comparison we care about is:

- Sprint 1 baseline / Sprint 2 best model
- Sprint 3 multi-task models

We want to look at:

- examples that are fixed by the multi-task setup
- examples that are broken by the multi-task setup
- whether the auxiliary `brevity` signal seems to help on summaries that are
  missing key points or compressing the source badly

The helper script `src/analyze_models.py` prints a few fixed and broken dev
examples so we can use them to write up a more detailed explanation.

We also compare a few simple rule-based ensembles in `src/ensemble_mtl.py` so
we can check whether the Sprint 2 and Sprint 3 models are complementary in a
useful way.

## Current Comparison

For the first iteration of the detailed analysis, we compared:

- the Sprint 1 traditional baseline
- the Sprint 3 traditional multi-task model

This is the most useful comparison right now because the traditional multi-task
model is the strongest Sprint 3 result on macro F1.

## Examples Fixed By The Multi-task Model

Some of the fixed examples are cases where the baseline predicted the majority
class `2`, but the multi-task model moved the prediction down to class `1`.
Examples include:

- `152_u`
- `159_u`

These are useful because they suggest the auxiliary `brevity` label may be
acting as a regularizer. In both cases, the summaries seem fairly complete, but
they are not clearly strong enough to deserve the highest `accuracy` label. The
multi-task model looks like it's not overly-crediting those summaries.

Another fixed example is:

- `45_s`

Here the baseline predicted class `1`, while the multi-task model moved it up
to the correct class `2`. This suggests the joint label can also help in the
other direction by supporting and correctly predicting summaries that are both 
fairly complete and accurate.

## Examples Broken By The Multi-task Model

Some broken examples include:

- `159_s`
- `184_u`

In these cases, the baseline correctly predicted class `2`, but the multi-task
model dropped the prediction to class `1`. A potential reason for this is that 
the model may be learning to be more cautious overall. That helps in some cases 
where overprediction is happening but it can also make the model undervalue 
 summaries that are actually prety good.

## Takeaway

The tuned Sprint 3 multi-task model seems to help predmoinantly making the
traditional model less overconfident about the top class while still keeping
more of the correct class `2` predictions than the first untuned version did.
This gives a better improvement in both accuracy and macro F1, even though it
ends up producing a few underpredictions. The overall effect still looks like
regularization rather than a massive behavioural change, which is a reasonable
outcome for this sprint.

The simple ensemble comparison suggests that the Sprint 2 and Sprint 3 models
are complementary in a good but still limited way. A confidence-gated hybrid that
uses the Sprint 3 prediction only when the multi-task model has a strong margin
reaches `0.750` accuracy and `0.415` macro F1. That's not the absolute best
macro-F1 result, but it is the best overall balance we found between the
highest accuracy Sprint 2 ensemble and the higher macro F1 Sprint 3 model.

We also believe it's imperative to keep the data itself in mind when interpreting these
results. This is still a relatively small dataset, the `accuracy` label is
imbalanced, and the annotations are pretty subjective because they reflect
human judgments of summary quality rather than the objective truth. Also, text data in 
general is messy, if you add that with the fact that these are ratings of summaries 
generated by an LLM, it becomes quite clear that high numbers would be virtually impossible. That means the task is not perfectly predictable, and some instability across models is expected. Overall, limited gains and some weird prediction changes are not necessarily signs of bad modeling but just a consequence of noisy low-resource data.
